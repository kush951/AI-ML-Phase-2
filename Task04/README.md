# PlaceMux — Match Explainability Engine

**Task 4 · Applications & Shortlisting · AI/ML Engineer · Week 2, Phase 2**

> Deliverable: *"Add match explainability."* Every match score PlaceMux shows a
> company must come with a plain-English reason — not a black box, not "trust me."

This is a complete, runnable, end-to-end implementation: synthetic-but-realistic
data → baseline → five competing models → best model selected on real metrics →
SHAP-based explainability → FastAPI serving layer → a working frontend console
that calls it live.

## 2. Quickstart

```bash
pip install -r requirements.txt

# 1. (Re)generate the sample dataset
python3 src/data_generation.py

# 2. Train, compare 5 models against the baseline, pick a winner, evaluate on held-out test
python3 src/train.py

# 3. Sanity check the explainer on a few rows
python3 src/explain.py

# 4. Start the API (from project root)
uvicorn src.api:app --reload --port 8000
# docs at http://localhost:8000/docs

# 5. Serve the frontend (separate terminal)
cd frontend && python3 -m http.server 8080
# open http://localhost:8080/index.html
```

The frontend is a static HTML file that calls the API at `http://localhost:8000`
(CORS is open in `src/api.py`). Both must be running for the live demo.

## 3. The data (Stage A.3 — sample data for this task)

`src/data_generation.py` builds a realistic, non-toy dataset, not a happy-path:

- **600 students** across 4 domains (Software, Data/AI, Design, Business), each
  with 4–9 verified skills. Verified scores are *noisy* — a student's "true"
  ability is latent and the assessment adds measurement noise, like a real
  verification quiz would.
- **120 jobs**, each with required *and* nice-to-have skills (different weights),
  a minimum-years bar, and a minimum verified-score bar.
- **4,000 student↔job pairs**, each with 12 engineered features (skill overlap
  counts/ratios, average and minimum verified score on required skills, domain
  match, experience gap, etc.).
- The match label is generated from a **latent quality function plus ~8% label
  noise** — so no model can hit 100%, exactly like a real hiring-quality label
  collected from human reviewers or downstream hire/no-hire outcomes.

This directly avoids the pitfall called out in the study guide: *"it only works
on a toy example, never on real sample data."*

## 4. Models compared (Stage B — multiple models, best one selected with numbers)

**Baseline first** (per the study guide's "Baseline first" principle): rank by
required-skill overlap ratio alone — no verified score, no domain, no experience.
Every later number is judged against this.

Five model families were trained on an identical 60/20/20 train/val/test split
(stratified, `random_state=42`), with a `StandardScaler` for the distance/linear
models:

| Model | Precision | Recall | F1 | FPR | ROC-AUC |
|---|---|---|---|---|---|
| **Baseline** (skill-overlap rule) | 0.933 | 0.379 | 0.539 | **0.023** | — |
| Logistic Regression | 0.859 | 0.714 | 0.780 | 0.099 | **0.849** |
| **Random Forest ← selected** | 0.858 | 0.722 | **0.784** | 0.102 | 0.846 |
| Gradient Boosting | 0.858 | 0.725 | 0.786 | 0.102 | 0.841 |
| SVM (RBF) | 0.855 | 0.722 | 0.783 | 0.104 | 0.822 |
| k-NN (k=25) | 0.860 | 0.703 | 0.774 | 0.097 | 0.834 |

*(full numbers in `reports/experiment_log.csv`, regenerated by `train.py`)*

### Selection rule (declared before touching the test set)

```
selection_score = 0.6 × ROC-AUC + 0.4 × F1      (on the validation set)
```

Random Forest and Gradient Boosting were within noise of each other; **Random
Forest was selected** for the best combined ranking quality and F1, and because
tree ensembles give cleaner, more stable SHAP explanations than boosting with
this feature set. The test set was touched **exactly once**, after the model
was already locked in:

| | Accuracy | Precision | Recall | F1 | FPR | ROC-AUC |
|---|---|---|---|---|---|---|
| Baseline (held-out test) | 67.9% | 0.904 | 0.335 | 0.489 | 0.030 | — |
| **Random Forest (held-out test)** | **80.3%** | 0.843 | 0.700 | **0.765** | 0.111 | 0.818 |

The selected model **beats the baseline by +12.4 points of accuracy and nearly
doubles F1** (0.489 → 0.765) on data it never saw during tuning. The baseline's
precision looks good only because it almost never says "match" (recall 0.34) —
exactly the kind of single-number-no-baseline trap the study guide warns about.

### Segment breakdown (no hidden weak spot)

Per the study guide: *"Quoting one accuracy number with no baseline and no
breakdown by segment"* is a pitfall. Held-out test, broken out by job domain
(`reports/segment_breakdown.csv`):

| Domain | n | Accuracy | Precision | Recall | FPR |
|---|---|---|---|---|---|
| Business | 284 | 79.2% | 0.850 | 0.637 | 0.088 |
| Data/AI | 171 | 80.1% | 0.831 | 0.753 | 0.151 |
| Design | 166 | 81.3% | 0.869 | 0.697 | 0.089 |
| Software | 179 | 81.0% | 0.824 | 0.744 | 0.134 |

No domain collapses relative to the overall number — the model generalises
across segments, not just on average.

## 5. Explainability (Stage B.4 — the core deliverable)

`src/explain.py` wraps the trained model in **SHAP** (`TreeExplainer` for the
Random Forest), so every prediction ships with:

1. A **match score** (0–100).
2. A **GOOD MATCH / WEAK MATCH** verdict.
3. A ranked list of **feature contributions** (SHAP values) — how much each
   signal pushed the score up or down, in the original units (e.g. "average
   verified score on required skills: 71.6 → +0.09").
4. A **plain-English sentence** built from the top contributors, e.g.:

   > *"This candidate looks like a strong match (92/100). Main reasons in
   > favour: average verified score on required skills (40.3); field match
   > (1); share of required skills the student has (0.8)."*

This is real per-instance SHAP, not a fixed template — the explanation changes
with the actual prediction, and the same machinery batches efficiently for
ranking a whole shortlist (`explain_batch`, used by `/shortlist/{job_id}`).

## 6. API (Stage C.1 — wired into an end-to-end flow)

`src/api.py`, FastAPI, served via `uvicorn`:

| Endpoint | Purpose |
|---|---|
| `GET /health` | which model is loaded |
| `GET /students`, `GET /jobs` | catalog data for the UI dropdowns |
| `GET /match/{student_id}/{job_id}` | **one explainable match** — score + SHAP breakdown + sentence |
| `GET /shortlist/{job_id}` | rank all students against one job — the company-facing view |
| `GET /metrics` | full experiment log + baseline vs. best-model test metrics |
| `GET /metrics/segments` | per-domain test breakdown |
| `GET /metrics/experiments` | raw per-model validation table |

Swagger docs at `/docs`.

## 7. Frontend (integrated, calls the live API)

`frontend/index.html` — the **"Explainable Match Console"** — a single static
HTML/JS file (no build step) styled as a verification ledger, fetching
everything live from the API:

- **Single match** mode: pick a student + job → a "stamp" card (GOOD MATCH /
  WEAK MATCH, score) and a **contribution ledger** — a credit/debit-style bar
  per feature, exactly mirroring the SHAP values from `/match/...`.
- **Shortlist** mode: pick a job → ranked table of every candidate with score
  + one-line reason, from `/shortlist/{job_id}`.
- **Model bake-off** table and **per-domain segment** cards, pulled live from
  `/metrics` and `/metrics/segments` — so the evaluation evidence is in the
  same screen as the live demo, not a separate slide.

This is the artifact for the Stage C.4 "2-minute live demo on real data."

## 9. Pitfalls from the study guide, and how this avoids each one

- *"The model is a black box, just trust it"* → every score ships with a SHAP
  breakdown and a sentence, not a bare number.
- *"Quality described with no numbers and no baseline"* → §4 has the full
  baseline-vs-5-models table, validation **and** held-out test.
- *"Only works on a toy example"* → §3 — noisy labels, 4,000 real-shaped pairs,
  best model tops out at 80% accuracy, not 100%.
- *"Tuning to the demo dataset until it collapses on real data"* → the
  selection rule was declared before touching the test set, and the test set
  was evaluated exactly once (`train.py`, §4).
- *"One accuracy number, no segment breakdown"* → §4 segment table by domain.
- *"Shipping a model you cannot explain"* → §5, the whole point of this task.

## 10. Honest limitations / what's next

- The dataset is synthetic — generated with a declared latent-quality function
  (visible in `data_generation.py`), not real PlaceMux production data. The
  pipeline (baseline → multi-model comparison → SHAP explainability → API →
  UI) is built to drop real verified-skill and JD data in without changing
  the architecture.
- "Upstream dependency: Ranking outputs" (per the study guide) — this task
  consumes verified skill scores as if ranking had already validated them;
  swap `data/students.json` / `data/jobs.json` for the real ranking-team
  output when it's ready.
- Search speed at marketplace scale (one of the self-check questions) isn't
  covered here — this task is about explainability, not retrieval. The
  `/shortlist` endpoint currently scores every student per job (fine at
  hundreds of students; would need an ANN/embedding pre-filter — see "Go
  deeper" in the study guide — before scoring at marketplace scale).
- Threshold for GOOD/WEAK MATCH is a fixed 0.5 on predicted probability; in
  production this should be tuned per the company's tolerance for false
  positives vs. false negatives, using the precision/recall trade-off curve.
